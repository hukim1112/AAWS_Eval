# 🧠 Agentic AI Hands-on: 핵심 교훈 및 발전 방향

이틀간의 멀티 에이전트 파이프라인(수집 → 분석 → 시각화)을 구축하며 얻은 중요한 아키텍처적 통찰(Insights)입니다. 

---

## 💡 Lesson 1: 에이전트는 '실행기'가 아닌 '전략가'여야 한다
* **관찰**: 에이전트에게 "페이지에서 데이터를 수집해"라고만 하면, 단순히 눈에 보이는 'Next' 버튼만 무한히 클릭하려 합니다.
* **해결**: 무작정 DOM을 탐색하기 전에, **우선 검토해야 할 전략 풀(Strategy Pool)**을 프롬프트로 내재화했습니다. (예: 네트워크 탭 분석, URL 패턴 분석 등)
* **시사점**: 에이전트 간 협업 시, 전달되는 메시지에는 단순한 행동 지침뿐만 아니라 **"어떤 전략을 채택했고, 그 이유는 무엇인지"**라는 메타 인지적 사고가 반드시 포함되어야 퍼포먼스가 기하급수적으로 상승합니다.

## 💡 Lesson 2: 도구가 곧 에이전트의 지능이다 (Tool Design = Intelligence)
* **관찰**: 프롬프트를 아무리 정교하게 작성해도 에이전트의 행동에는 한계가 있었습니다.
* **해결**: Navigator에게 `get_page_structure`를 주어 분석가로 만들고, Coder에게 `run_command`를 주어 개발자로 만들었습니다. Analyst에게 `create_chart`를 주니 데이터 사이언티스트가 되었습니다.
* **시사점**: 같은 LLM이라도 **어떤 도구를 주느냐에 따라 완전히 다른 에이전트가 됩니다.** 프롬프트는 에이전트의 '성격'을 정하지만, 도구는 에이전트의 '능력'을 결정합니다. 도구의 입출력과 Docstring 설계가 성능의 핵심입니다.

## 💡 Lesson 3: 프롬프트 엔지니어링을 넘어 '컨텍스트 엔지니어링'으로
* **관찰**: 시스템 프롬프트에 모든 예외 상황과 지시사항을 욱여넣자, 토큰 낭비가 심해지고 모델의 주의력이 분산되었습니다.
* **해결**: 정적 프롬프트 대신 런타임에 **미들웨어(Middleware)**를 활용해 필요한 순간에만 정보를 주입했습니다. (예: 작업 직전 과거 메모리 주입, 코드 실행 시 파일 목록 주입)
* **시사점**: "무엇을 쓸까(Prompt)"보다 **"해당 작업에 필요한 정보를, 적절한 시점에, 적절한 양만큼 언제 전달할까(Context)"**가 훨씬 중요합니다. 프롬프트를 늘리지 말고 미들웨어 파이프라인을 정교하게 설계하세요.

## 💡 Lesson 4: 에러를 통한 지속적 학습 (Continuous Learning)
* **관찰**: 모든 사이트의 방어 기제나 독특한 DOM 구조를 사전에 프롬프트에 담는 것은 불가능합니다.
* **해결**: 에이전트가 실패와 재시도를 거쳐 우회법을 찾아내면, 이를 휘발시키지 않고 **패턴 메모리(Pattern Memory)**에 저장하여 다음 작업에 RAG로 주입했습니다.
* **시사점**: 시스템은 경험을 통해 진화해야 합니다. 한 번 겪은 에러와 해결책을 시스템의 자산으로 축적하는 구조를 만들어야 진정한 '자율형' 에이전트가 완성됩니다.

## 💡 Lesson 5: 결정론(코드)과 비결정론(LLM)을 연결하는 하이브리드 인터페이스
* **관찰**: 에이전트 시스템은 본질적으로 "두 개의 뇌"가 협력해야 합니다. 파이썬 코드(결정론)는 정확한 Key와 데이터 타입을 요구하며, LLM(비결정론)은 의도와 맥락 같은 풍부한 자연어를 요구합니다. 이 둘을 분리해서 관리(예: 지시는 txt, 채점 기준은 JSON)하면 동기화가 어긋나 파이프라인이 쉽게 붕괴됩니다.
* **해결**: 파일 상단에는 파이썬 코드가 파싱할 엄격한 평가 기준(YAML Frontmatter)을, 하단에는 LLM이 읽을 자연어 지시사항(Markdown)을 결합한 **하이브리드 문서 구조(`scenarios/*.md`)**를 채택했습니다.
* **시사점**: 통제 불가능해 보이는 AI를 기존 소프트웨어 공학의 통제권 안으로 끌어들이려면 **기계와 LLM이 완벽히 동기화된 '단일 진실 공급원(Single Source of Truth)'**이 필요합니다. '무엇을 할 것인가(명령)'와 '어떻게 채점할 것인가(규격)'를 분리하지 말고 하나의 원자(Atom) 단위로 결합하세요.

## 💡 Lesson 6: 프롬프트는 코드다 (Prompt-as-Code)
* **관찰**: 워크플로우 파이썬 코드 안에 긴 프롬프트를 하드코딩해 두었더니, 프롬프트 수정과 버전 관리가 매우 고통스러웠습니다.
* **해결**: 프롬프트를 `src/prompts/` 패키지로 독립시키고 모듈화했습니다.
* **시사점**: 프롬프트는 일회성 텍스트가 아닌 **소프트웨어의 핵심 컴포넌트**입니다. 코드와 동일한 수준으로 모듈화하고, 역할별로 분리하여 버전 관리를 해야 팀 협업과 실험 속도가 올라갑니다.

## 💡 Lesson 7: 하네스 엔지니어링 (Evaluation-Driven Development)
* **관찰**: 에이전트를 만들고 "잘 되는 것 같다"는 직관에만 의존하면, 프롬프트나 도구를 변경했을 때 전체 시스템에 미치는 영향을 파악하기 어렵습니다.
* **해결**: 시나리오(테스트 케이스), 평가 기준, 자동 실행기, 그리고 LLM-as-a-Judge(판정자)를 결합한 **에이전트용 테스트 하네스(AAWS_Eval)**를 구축했습니다.
* **시사점**: 에이전트의 결과물은 하나의 정답이 아니라 '범위(Spectrum)'를 가집니다. 따라서 기존의 TDD(Test-Driven Development)는 에이전트 시대에 **\"평가 주도 개발(EDD)\"**로 진화해야 합니다. 합격 기준을 먼저 정의하고 측정 가능한 환경 위에서 개선 사이클을 돌리세요.

## 💡 Lesson 8: 시스템 프롬프트는 '패치'가 아니라 '원칙'이다 (Generalization over Overfitting)
* **관찰**: GitHub Trending 시나리오에서 Coder가 "25개 수집" 목표에 13개만 얻자, 코드를 무한 수정하다 Recursion Limit에 도달했습니다. 이를 해결하기 위해 시스템 프롬프트에 "25개 기대, 13개 수집 같은 경우엔 정상으로 인정하라"는 예외 규칙을 추가했으나, 이는 하나의 사례에 과적합(Overfitting)된 패치일 뿐이었습니다.
* **해결**: 특정 숫자나 사례를 제거하고, 범용적 의사결정 원칙으로 대체했습니다:
  - **Coder에게**: "재시도 전에 원인을 분류하라 — 내 코드 버그인가, 외부 요인인가? 후자라면 가설과 함께 즉시 보고하라."
  - **Supervisor에게**: "Coder가 외부 요인을 의심하면, 재시도를 강요하지 말고 Navigator에게 실제 상태를 확인시킨 뒤 판단하라."

* **실제 프롬프트 Before / After 비교**:

  > **Coder 프롬프트 — 과적합 버전 ❌**
  > ```
  > 데이터 수집 개수 부족(예: 25개 기대, 13개 수집)과 같은 검증 오류의 경우,
  > 대상 페이지의 실제 데이터 양이 적어서 발생하는 문제일 수 있습니다.
  > 이 때는 무의미한 코드 수정을 멈추고 즉시 작업을 중단하세요.
  > ```
  > 📌 문제: 숫자(25, 13)가 GitHub Trending 사례를 그대로 반영. 원인을 "페이지 데이터 부족"으로 단정하여, 진짜 코드 버그인 경우에도 성급하게 포기하게 만듦.

  > **Coder 프롬프트 — 범용 원칙 버전 ✅**
  > ```
  > 에러나 검증 실패가 발생하면, 재시도 전에 반드시 원인을 분류하세요:
  >   a) 내 코드의 버그인가? → 코드를 수정하고 재시도. (최대 3회)
  >   b) 코드 외부의 요인인가? → 코드 수정은 무의미하므로 즉시 중단하고 보고.
  > 같은 수정을 반복하거나, 코드를 고쳐도 결과가 변하지 않는다면 외부 요인을 강하게 의심하세요.
  > ```
  > 📌 개선: 특정 숫자 제거. "원인 분류"라는 사고 프레임워크를 제공하여 어떤 종류의 실패에도 적용 가능.

  > **Supervisor 프롬프트 — 과적합 버전 ❌**
  > ```
  > Coder가 "페이지에 실제 항목 개수가 부족하다"며 데이터 개수 불일치
  > (예: 25개 목표이나 13개 수집)를 보고할 경우, 이를 실패로 간주하여
  > 강제로 재시도하지 말고 정상 수집된 것으로 인정하여 사용자에게 보고하세요.
  > ```
  > 📌 문제: "정상으로 인정하라"고 답을 하드코딩. Supervisor가 Navigator에게 확인하는 선택지 자체를 제거함.

  > **Supervisor 프롬프트 — 범용 원칙 버전 ✅**
  > ```
  > Coder가 "코드는 정상이나 외부 요인이 의심된다"고 보고하면,
  > Coder에게 재시도를 강요하지 말고 Navigator에게 mode="chat"으로
  > 해당 페이지의 실제 상태를 확인시키세요. 확인 결과를 바탕으로 최종 판단하세요.
  > ```
  > 📌 개선: 판단을 고정하지 않고, "크로스 에이전트 검증 후 판단"이라는 의사결정 절차를 제공.

* **시사점**: 시스템 프롬프트는 ML 모델의 학습과 같습니다. 특정 실패 사례를 그대로 프롬프트에 넣으면 **과적합**되어 그 케이스만 해결하고 다른 상황에서 오작동합니다. 반대로 실패에서 **일반 원칙을 추출**하여 반영하면, 경험하지 못한 새로운 상황에도 대응하는 **범용성(Generalization)**을 얻습니다. 버그를 고칠 때 항상 스스로에게 물으세요 — *"이 수정은 이 한 건만 고치는 패치인가, 아니면 유사한 모든 상황을 커버하는 원칙인가?"*

## 💡 Lesson 9: 프롬프트의 3계층 분리와 마이크로매니지먼트 제거 (Separation of Concerns)
* **관찰**: 에이전트의 실수를 방지하고자 시스템 프롬프트에 구체적인 도구 파라미터명, 특정 사이트 예시(다나와/쿠팡), CSS 셀렉터 작성법(tutorial), 심지어 권장 라이브러리(BS4 vs Playwright)까지 전부 기재했습니다. 결과적으로 프롬프트가 매뉴얼처럼 변질되어 토큰을 낭비하고, 모델이 자체 추론 능력을 발휘하지 못하는 **'지시 수용 모드'에 갇히는 현상**이 발생했습니다.
* **해결**: 정보의 성격에 따라 프롬프트를 3계층으로 완전히 분리(Decoupling)했습니다.
  1. **Layer 1: Identity & Principles (시스템 프롬프트)**
     - "당신은 전략가다", "에러 발생 시 원인을 분류하라" 등 거의 변하지 않는 역할과 **판단 원칙**만 남겼습니다.
  2. **Layer 2: Tool Contracts (도구 Docstring)**
     - `mode="chat"`이나 `wait_seconds=5` 같은 구현 디테일은 프롬프트에서 모두 제거하고, 해당 도구의 `__doc__` 안으로 이동시켰습니다. (도구 시그니처 변경 시 프롬프트를 수정할 필요가 없어짐)
  3. **Layer 3: Task Context (런타임 주입)**
     - 특정 사이트의 특성이나 이번 작업의 제약 사항은 시나리오 파일(`*.md`)에서 런타임에 동적으로 주입하도록 설계했습니다.
* **시사점**: 시스템 프롬프트에 너무 많은 제약과 구체적인 방법을 적는 것(마이크로매니지먼트)은 오히려 LLM의 가장 큰 무기인 '유연한 추론 능력'을 억압합니다. **프롬프트에는 '무엇을(What)'과 '왜(Why)'라는 원칙만 남기고, '어떻게(How)'는 도구 명세와 스키마에게 양보하세요.** 프롬프트 다이어트는 비용 절감뿐 아니라 지능 향상의 핵심 열쇠입니다.

## 💡 Lesson 10: 정보의 압축과 다단계 분석 원칙 (Data Abstraction & Protocol)
* **관찰**: 다나와 사이트 분석 시 Navigator가 광고 영역을 실제 상품으로 착각하여 외부 사이트(bridge)로 이탈하고, 페이징 로직 파악에 실패했습니다. 실패 보고서의 제안 중 "파이썬 코드에 `.ad_product` 등 광고 클래스를 하드코딩하여 미리 제거하자"는 의견이 있었으나, 이는 **Lesson 8에서 경계한 전형적인 '과적합(Overfitting) 패치'**입니다. 타 사이트에서는 정상 클래스일 수 있기 때문입니다.
* **해결**: 특정 사이트에 종속된 하드코딩을 배제하고, 아키텍처적 관점에서 범용적인 솔루션을 도출했습니다.
  1. **정보 압축 (도구 레벨 - `extract_dom_skeleton`)**: 300KB의 Raw HTML을 통째로 던져주는 대신, 태그/클래스/자식 수 정보만 남긴 압축된 DOM 트리 맵(~5KB)을 제공. LLM은 수많은 텍스트보다 반복되는 트리 구조의 미세한 차이(예: 광고 노드와 실제 상품 노드의 구조적 차이)를 파악하는 데 훨씬 탁월합니다.
  2. **도구 기능 확장 (결정론적 추출 - `verify_selectors::attr()`)**: 셀렉터 검증 도구에 `::attr(href)`, `::attr(src)` 등 속성 추출 구문을 추가하여, 링크 URL이나 이미지 경로 같은 속성값을 결정론적으로 읽을 수 있게 했습니다. 이를 통해 에이전트가 단순 데이터 읽기를 위해 비용이 큰 `browse_web`(LLM 기반 브라우저 조작)에 의존하는 빈도를 구조적으로 줄였습니다. **핵심 통찰: 프롬프트를 고치는 것이 아니라, 결정론적 도구의 기능을 확장하여 비결정론적 도구의 사용을 줄이는 것이 더 안정적입니다.**
  3. **절차적 엄격성 (프롬프트 레벨 - 다단계 분석 프로토콜)**: "목록을 완전히 검증(광고 구분, 페이징 파악, 실제 상세 URL 확보)하기 전에는 절대 상세 페이지 분석으로 넘어가지 마라"는 절차적 가이드를 추가했습니다.
  4. **예외 전달 규격화 (스키마 레벨 - `PageLayer` 보강)**: `pagination_details`(JS 호출방식 등 세부구현) 및 `exclude_selectors`(광고/예외 필터) 필드를 스키마에 추가하여, Navigator가 파악한 예외 상황을 Coder에게 유실 없이 전달하도록 규격화했습니다.
* **시사점**: LLM의 컨텍스트 윈도우가 커졌다고 해서 원시 데이터(Raw data)를 그대로 들이붓는 것은 "Garbage In, Garbage Out"을 초래합니다. **LLM이 가장 잘 추론할 수 있는 형태(구조화된 요약본)로 정보를 가공하여 전달하는 '데이터 인터페이스 설계'**가 모델의 지능을 극대화합니다.
